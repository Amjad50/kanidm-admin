# Setup — docker-compose full stack

End-to-end: kanidm + admin-ui + Caddy. Browse to
https://idm.localhost/admin when you're done.

This setup uses `idm.localhost` as the domain. Every modern OS resolves
`*.localhost` to 127.0.0.1 by default (RFC 6761) — you don't need to
edit `/etc/hosts`.

## Prerequisites

- Docker (24+) + `docker compose` v2
- ~5 GB free disk (kanidm + docker images)
- Ports `80` and `443` free on the host (Caddy listens here)

## Quick start

```bash
cd deploy

# 1. Make sure idm.localhost resolves to 127.0.0.1.
#    Most modern Linux distros do this automatically via nss-myhostname, but
#    not all of them. Check with:
#      getent hosts idm.localhost
#    If that prints nothing, add the entry yourself:
#      echo "127.0.0.1   idm.localhost" | sudo tee -a /etc/hosts
#    macOS, Windows: browsers usually handle *.localhost without /etc/hosts,
#    but it doesn't hurt to add it explicitly.

# 2. (Optional — no-op for now.) `./bootstrap.sh init` is a stub kept for
#    forward compatibility; it will generate an OAuth2 cookie signing key
#    once OAuth2 consent lands in master. Safe to skip today.

# 3. Build the admin-ui image and bring everything up.
docker compose up -d --build

# 4. Wait ~30s for kanidm to finish first-boot, then recover the admin password.
./bootstrap.sh passwords

# 5. Trust Caddy's internal CA (one-time; see "Cert trust" below).

# 6. Browse to https://idm.localhost/admin to reach the admin UI. The bare
#    https://idm.localhost/ goes to kanidm's own UI — admin-ui only owns
#    the /admin/* tree.
```

That's it.

## Cert trust (one-time)

Caddy generates a local CA on first boot. Until your machine trusts
that CA, browsers will warn on every visit. Two options:

### Option A: Install the CA system-wide (Linux, recommended)

```bash
docker compose exec caddy cat /data/caddy/pki/authorities/local/root.crt \
    | sudo tee /usr/local/share/ca-certificates/caddy-local-root.crt > /dev/null
sudo update-ca-certificates
```

Firefox keeps its own store and won't see system CAs by default. To
import there too, in Firefox: `about:preferences#privacy` → "View
Certificates" → "Authorities" tab → "Import" → select the file at
`/usr/local/share/ca-certificates/caddy-local-root.crt` → check
"Trust this CA to identify websites".

Chrome / Brave / Edge use the system trust store on Linux — they'll
work after `update-ca-certificates`.

### Option B: Click-through (lazy)

Just visit https://idm.localhost/, accept the browser warning, mark
the cert as trusted. Each browser does this slightly differently.
WebAuthn / passkey registration is more reliable with Option A
because some browsers refuse to invoke the platform authenticator
against an untrusted origin.

## What runs

```
                  ┌─────────────────────────┐
  https://443 ──→ │  caddy:2.10 (TLS term)  │ ──┬─→ admin-ui:3000 (this project)
                  │  cert: internal CA      │   │
                  └─────────────────────────┘   └─→ https://kanidm:8443 (kanidm/server:1.10.0)
```

- **caddy**  — TLS termination, path-based routing per
  `deploy/caddy/Caddyfile`. Routes `/admin/*` (the whole admin UI —
  pages, login, /me, healthz, static assets — all live under that
  one prefix) to `admin-ui:3000`; everything else (root, `/v1/*`,
  `/scim/*`, `/ui/*`, `/oauth2/*`, `/.well-known/*`, etc.) falls
  through to `kanidm:8443`.
- **kanidm** — official `kanidm/server:1.10.0` image. Stores db in
  the `kanidm_data` volume. TLS cert in the `kanidm_certs` volume,
  generated by the `kanidm-init` service on first boot.
- **admin-ui** — built from this repo via `Dockerfile`. Reads
  `deploy/admin-ui/kanidm-admin-ui.toml`. Talks to kanidm over the
  internal docker network.
- **kanidm-init** — one-shot container, generates the self-signed
  cert kanidm needs for its internal `:8443`. Re-running is a no-op
  (it skips if the cert exists).

## Daily ops

| What | How |
|---|---|
| Start everything | `docker compose up -d` |
| Stop everything | `docker compose down` |
| Logs | `docker compose logs -f admin-ui` (or `kanidm` / `caddy`) |
| Reset kanidm db | `docker compose down -v` (deletes the volume — irreversible) |
| Regenerate passwords | `./bootstrap.sh passwords` |
| Rebuild admin-ui after code changes | `docker compose up -d --build admin-ui` |
| Shell into kanidm | `docker compose exec kanidm sh` |
| Run kanidm CLI | `docker compose exec kanidm kanidm <args>` |

## Production differences

This is a local-dev setup. For production:

1. **Domain.** Replace `idm.localhost` everywhere with your real
   hostname (`server.toml`, `Caddyfile`, `kanidm-admin-ui.toml`).
2. **TLS.** Drop `tls internal` from `Caddyfile`. Caddy will use
   Let's Encrypt automatically if your domain resolves publicly and
   ports 80/443 are reachable.
3. **Backups.** Kanidm's `online_backup` config in `server.toml` runs
   daily at 22:00 UTC. The output goes to the `kanidm_data` volume's
   `/data/backups` subdir. Copy it off regularly.
4. **Persistence.** All state is in named volumes (`kanidm_data`,
   `caddy_data`, `kanidm_certs`). `docker compose down` preserves
   them; `docker compose down -v` destroys them.

## Troubleshooting

**"connection refused" from admin-ui to kanidm:** kanidm is still
booting. The `depends_on: kanidm.service_healthy` should prevent this
in steady state; wait 30–60s after `docker compose up -d`.

**"version mismatch" in admin-ui logs:** the `kanidm_client` Rust
crate version in `Cargo.toml` doesn't match the running kanidm image
tag. Bump one to match the other; both default to `1.10.0` here.

**Browser warns "NET::ERR_CERT_AUTHORITY_INVALID":** the Caddy
internal-CA trust step (above) hasn't been done, or the browser uses
its own store. Either install the CA into the browser's store too
(Firefox) or click-through.

**`./bootstrap.sh passwords` says "could not parse password":**
kanidmd's output format changed in a newer version. Run it
interactively to see the raw output:
```
docker compose exec kanidm kanidmd recover-account -c /data/server.toml idm_admin
```

**Admin-ui returns 500 on startup:** check
`docker compose logs admin-ui` — the most likely cause is that kanidm
isn't reachable yet, or `deploy/admin-ui/kanidm-admin-ui.toml` has a
typo. Confirm kanidm is healthy with `docker compose ps`.

## What this is NOT

- Not a production-hardened setup. No log shipping, no monitoring, no
  resource limits.
- Not multi-replica. Both admin-ui and kanidm are single-instance.
- Not zero-trust. The admin-ui→kanidm connection uses
  `accept-invalid-certs` (the connection is internal-network-only,
  which is acceptable for a self-signed cert).
