# 05 — Login: Backup Code Entry

The user enters a recovery backup code as the second factor for the `passwordbackupcode` mechanism (or as a fallback from TOTP).

## Purpose

Collect a one-time backup code. Verify with kanidm. Complete authentication. Show a clear "this code is consumed" notice so the user knows their remaining-code count just decreased.

## Layout

Same centered card pattern (~440px wide).

```
              ┌────────────────────────┐
              │       Kanidm           │
              │   idm.example.com      │
              ├────────────────────────┤
              │  Enter a backup code   │
              │                        │
              │  ┌──┐                  │
              │  │AS│ Alice Smith      │
              │  └──┘ alice.smith@…    │
              │                        │
              │  Backup code           │
```

## Card content

**Header:** wordmark + domain.

**Title:** "Enter a backup code"

**Identity row:** avatar + name + SPN (same pattern as 03/04).

**Backup code input:**
- Label "Backup code"
- Type `text` (visible by default — backup codes are not as sensitive as a password in real-time, and visibility helps the user copy from a printout)
- Reveal toggle is optional. Designer can omit; if included, default to visible.
- Monospace font for the input (backup codes are typically hex/alphanumeric strings)
- Format hint placeholder: `xxxx-xxxx-xxxx` (kanidm backup codes follow a similar format)
- Accept input with or without dashes (UI normalizes before submitting)
- Autofocus

**Helper text:**
"Each code can only be used once. Make sure you have unused codes remaining."

**Primary action:** "Verify" button.

**Secondary link:** "← Choose another method".

## States

- **Idle:** as described.
- **Verifying:** input read-only, button spinner + "Verifying…".
- **Error — wrong code:** inline error: "That backup code is invalid or has already been used. Try a different code."
- **Error — no codes left:** denied screen (07) with reason "No backup codes remaining. Contact an administrator to reset your credentials." (kanidm would return a specific error.)
- **Success:** redirect to dashboard. A toast appears informing them: "Backup code used. You have 5 codes remaining."

## Sample data

- Identity: Alice Smith / `alice.smith@idm.example.com`
- Empty input placeholder: `xxxx-xxxx-xxxx`
- Filled input mockup: `a7b3-2c8d-9f1e`
- Toast after success: "Backup code used. You have 5 codes remaining."

## Edge cases

- **Pasted code with surrounding whitespace:** trim before submitting.
- **Code with or without dashes:** accept both. Normalize before sending to API.
- **All codes used:** kanidm denies with a specific error. UI shows denied screen with admin-contact message.
- **Codes regenerated by admin:** old codes invalidated. User sees "That code has already been used" for old ones — same error path.

## Tone

Helpful. The user is using a recovery option — they may be stressed (lost phone, etc.). The copy should be calming and clear. "Each code can only be used once" — direct, not condescending.
